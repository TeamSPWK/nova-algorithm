# Claude Filter — 프롬프트/에이전트 운영 진단 스킬

## 한 줄 요약

LLM에게 지시하는 산출물(프롬프트, CLAUDE.md, 스킬, 에이전트 설정)이 프로덕션 수준으로 작동하는지 5축으로 진단하고, 구체적 개선 방향을 제시한다.

## 왜 만들었나

2026년 3월 Claude Code 소스 유출 사건의 분석에서 반복적으로 드러난 교훈이 있다:

> 프로덕션 에이전트의 승패는 모델 성능이 아니라 하네스(컨텍스트 조립 + 도구/권한 + 검증/복구 + 메모리/압축)가 결정한다.

이 교훈을 "알고 있다"에서 "지키고 있다"로 전환하려면 정기적인 점검이 필요하다. Claude Filter는 그 점검 도구다.

## 자매 스킬과의 관계

```
pikes-filter  → 코드 복잡도 진단 (Rob Pike 5원칙)
claude-filter → 프롬프트/에이전트 운영 진단 (프로덕션 에이전트 설계 교훈)
```

pikes-filter가 "이 코드에 과잉 복잡성이 있는가?"를 묻는다면, claude-filter는 "이 지시 체계가 프로덕션에서 버틸 수 있는가?"를 묻는다.

## 진단 5축

### 축1: 지시 구조 (Instruction Structure)

**원리**: 프롬프트는 한 줄 주문이 아니라 운영 프로토콜이다.

"깔끔하게 작성해줘" 같은 추상 지시는 모델마다, 턴마다 해석이 달라진다. "2칸 들여쓰기, 함수당 50줄 이내"처럼 검증 가능한 지시만이 일관된 결과를 낳는다. Do(해야 할 것)와 Don't(하지 말 것)가 뒤섞이면 모델은 우선순위를 추측하게 되고, 추측은 실패의 시작이다.

**점검 핵심**:
- Do/Don't가 분리되어 있는가
- 각 규칙이 사람이든 코드든 검증할 수 있는가
- 필요한 곳에 숫자(제한)가 있는가

### 축2: 검증 루프 (Verification Loop)

**원리**: 에이전트의 품질을 올리는 가장 강력한 레버는 더 똑똑한 모델이 아니라 검증 루프를 강제하는 설계다.

"완료"를 모델의 자기 선언에 맡기면, 모델은 해피 패스만 통과해도 끝났다고 판단한다. 프로덕션 에이전트는 "생성 → 검증 → 수정 → 재검증" 루프를 구조적으로 강제한다. 검증을 못했으면 "완료"라고 쓰지 않는 것 자체가 설계 원칙이다.

**점검 핵심**:
- "완료"가 검증 통과 조건으로 정의되어 있는가
- 검증 실패 시 행동이 정의되어 있는가
- 해피 패스만이 아닌 경계/실패 조건을 다루는가

### 축3: 컨텍스트/비용 설계 (Context & Cost)

**원리**: 턴은 대화 단위가 아니라 비용/성능/리스크 단위다.

에이전트는 도구 호출을 포함한 반복 루프(턴)로 동작하며, 복잡한 작업은 수십 번의 턴이 이어진다. 턴 상한과 예산 상한 없이 돌리면, 잘못된 프롬프트 하나가 비용 폭발을 일으킨다. 변하지 않는 부분(규칙, 스키마)과 매 턴 바뀌는 부분을 분리하면 캐시 효율이 올라간다.

**점검 핵심**:
- 턴/예산 제한이 있는가
- 정적/동적 프롬프트가 분리되어 있는가
- 컨텍스트 오버플로우 대응 전략이 있는가

### 축4: 메모리/상태 관리 (Memory & State)

**원리**: 상태 외부화의 목적은 단순 기억이 아니라 디버깅 가능성과 비용 통제 가능성이다.

규칙을 매번 프롬프트에 복붙하면 세션마다 누락이 생기고, 파일로 외부화하면 일관성이 보장된다. 단, 메모리 파일이 비대해지면 컨텍스트를 잡아먹고 준수율이 떨어진다. 죽은 규칙이 쌓이면 노이즈가 신호를 덮는다.

**점검 핵심**:
- 규칙이 파일로 지속화되어 있는가
- 메모리/설정 파일이 200줄 이내인가
- 죽은 규칙/오래된 메모리가 쌓여 있지 않은가

### 축5: 권한/도구 게이팅 (Permission & Tool Gating)

**원리**: 에이전트의 핵심 루프는 단순할 수 있지만, 하네스는 복잡하고 정교해야 한다.

권한을 전역 스위치로 관리하면, 읽기만 해도 되는 작업에서 삭제 권한까지 열린다. 도구 단위로 게이트하고, 읽기(안전)와 쓰기/삭제/배포(위험)를 구분하면 속도와 안전을 동시에 잡는다. 고자율 실행은 "프롬프트만 세게"가 아니라 실행 직전의 추가 심사(가드/분류기)와 함께 설계해야 한다.

**점검 핵심**:
- 도구별 허용/차단이 있는가
- 읽기/쓰기 구분이 있는가
- 단계적 자율성(plan → edit → auto)이 있는가

## 사용법

```
/claude-filter                    # 대화에 붙여넣은 프롬프트를 진단
/claude-filter path/to/CLAUDE.md  # 파일을 지정해 진단
```

## 출력 예시

```
## 1. 대상 식별
medium-engine 프로젝트의 CLAUDE.md. 설계엔진 모듈의 개발 규칙 및 운영 지침.

## 2. 스코어카드
| 축                    | 판정 | 핵심 발견                                      |
|-----------------------|------|-------------------------------------------------|
| 1. 지시 구조          | PASS | Do/Don't 명확 분리, 수치 제약(200줄) 포함        |
| 2. 검증 루프          | WARN | 벤치마크 있으나 완료 게이트가 암묵적              |
| 3. 컨텍스트/비용      | WARN | 문서 인덱스 있으나 턴/예산 제한 미정의            |
| 4. 메모리/상태        | WARN | 메모리 70+개, 절반 이상 과거 실험 기록            |
| 5. 권한/도구          | PASS | 배포/SSH/push 도구별 게이트 명시                  |

## 3. 개선 액션
1. 메모리 위생: 최근 1개월 미참조 메모리를 아카이브 분리 — 노이즈가 신호를 덮고 있음
2. 완료 게이트 명시: "벤치마크 FAR 회귀 0건" 을 커밋 전 하드 게이트로 추가
3. 턴 정책 추가: 장기 최적화 사이클의 턴/예산 상한 명시

## 4. 리라이트 제안
개선 사항을 적용한 버전을 만들까요?
```

## 설계 원칙

1. **문제점부터 말한다** — 칭찬은 생략, 인사말 없음
2. **근거를 붙인다** — 모든 판정에 인용 또는 부재 지적
3. **즉시 실행 가능한 액션만** — "전면 재설계"가 아닌 "지금 바로 할 수 있는 것"
4. **진단 자체가 가벼워야 한다** — 스킬이 스스로 축4(고정비용)를 위반하면 안 됨

## 배경 자료

이 스킬의 5축은 다음 교훈에서 도출되었다:

- 프롬프트는 운영 프로토콜이다 → 축1
- 검증 루프를 강제하는 설계가 모델 성능보다 낫다 → 축2
- 턴은 비용/리스크 단위다 → 축3
- 상태 외부화는 디버깅 가능성이다 → 축4
- 하네스의 권한/도구 설계가 성패를 가른다 → 축5
