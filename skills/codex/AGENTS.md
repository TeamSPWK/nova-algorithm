## 1. Root cause 우선

증상이 아니라 원인을 고친다. band-aid·임시 우회는 부채를 키운다.

- 에러를 `try/except`로 삼키지 않는다. 잡았으면 의미 있게 다루거나 다시 던진다.
- 안전 체크를 우회하지 않는다 (`--no-verify`, 검증 skip, hook bypass 등 금지).
- 테스트가 깨졌으면 테스트가 아니라 코드부터 의심한다.
- 원인을 모를 때 추측 패치하지 않는다. 멈추고 가설·검증 결과를 보고.

> 임시 대응이 불가피하면 반드시 **"임시"임을 명시** + 근본 해결책 함께 제시.

## 2. Pike 원칙 (데이터·단순 우선)

> "Data dominates. ... Fancy algorithms are slow when n is small, and n is usually small."  
> "When in doubt, use brute force."

코드 작성/리뷰 시 다섯 질문을 통과해야 한다.

| 원칙 | 질문 |
|------|------|
| Premature Optimization 금지 | 프로파일링 증거 없이 직관으로 최적화하지 않았는가? |
| Complexity vs N | 작은 N에 과한 알고리즘을 쓰지 않았는가? |
| Algorithm Overkill 회피 | brute force·greedy·직관적 loop으로 풀 수 있는데 복잡하게 쓰지 않았는가? |
| Logic/Data 불균형 해소 | 데이터 구조를 보완하면 if/else·깊은 loop을 줄일 수 있지 않은가? |
| Missing Lookup | 런타임 계산 대신 lookup table·map·정적 데이터로 치환 가능한가? |

핵심: **잘 설계된 데이터 구조 > 영리한 로직**.

## 3. YAGNI / 변경 최소화

요청 범위를 넘는 변경 금지.

- 미래 확장성을 가정한 추상화·인터페이스·플러그인 시스템을 미리 만들지 않는다.
- 요청 외 리팩토링 금지. 옆에 보이는 냄새는 별도 task로.
- feature flag·shadow 토글·점진 롤아웃 금지. 새 코드는 바로 ON으로 머지.
- "혹시 모르니까" 옵션·파라미터·구성값을 추가하지 않는다. 필요해지면 그때 추가.

> 작은 PR·작은 변경·되돌리기 쉬운 구조가 원칙.

---

## 4. Simple > Clever

읽기 쉬운 코드가 우선.

- 직관적 이름. `disposition` 대신 `status`, `ledger` 대신 `history`, `origin` 대신 `source`.
- 작은 함수, 단일 책임. 이름이 안 떠오르면 함수가 너무 큰 신호.
- 영리한 한 줄보다 읽히는 세 줄. 한 번 쓰고 다시 안 보는 코드는 없다.
- 주석은 "왜"만. "무엇"은 코드가 말한다. 죽은 코드 주석으로 남기지 말고 삭제.
- 일관된 스타일 > 개인 취향. 기존 코드 컨벤션 우선.

---

## 5. 실 환경 검증

테스트가 통과해도 실측이 아니면 통과 아니다.

- mock으로 회귀를 가리지 않는다. 실 DB·실 외부 API·실 데이터로 검증.
- 통합 테스트·E2E·실 파이프라인 1회 통과를 unit test 100개보다 신뢰.
- "코드 존재 ≠ 동작". 빌드·실행·결과 확인까지 끝나야 완료.
- 환경 변경(env var, 설정 파일, 마이그레이션)은 3단계 확인: **현재 상태 → 변경 → 반영 검증**.

## 작업 종료 시 자체 점검

코드 변경 후 위임자에게 결과를 넘기기 전 다음 5문항 자가 체크:

1. 증상 패치인가, 원인 수정인가?
2. Pike 5원칙 중 어긴 것이 있는가?
3. 요청 범위 밖 변경이 섞였는가?
4. 동일한 의미를 더 간단히 표현할 수 있는가?
5. 실제 환경에서 검증했는가, 아니면 "통과할 것이다" 가정인가?
